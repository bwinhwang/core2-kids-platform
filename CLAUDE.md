## 1. 一句话定位 & 目标用户

孩子双手端着 M5Stack Core2,**左右前后倾斜机身**,屏幕上一个可爱小角色(球)就跟着重力滚动,
**滚进「家」(目标点)就触发一场庆祝**。然后自动进入下一个略难一点的迷宫。

- **目标用户:3~4 岁幼儿**(不识字、手部精度低、易过度操作、注意力短)。
- **核心体验:即时因果**——倾斜→立刻看到球动、听到声、感到震动。无规则文字、无失败惩罚、无计时压力。
- **唯一"输入"是倾斜**(IMU),不依赖触摸精度、不依赖按键。触摸只用于家长隐藏菜单。

### 设计第一性原则(贯穿全文)

1. **零失败**:撞墙不是"输",只是被温柔弹开 + 一个搞笑反馈。永远只有"还没到"和"到了"。
2. **即时且可无限重复**:任何动作 < 100ms 内有反馈;一关随便玩多久、来回滚都行。
3. **多通道冗余反馈**:同一事件同时给 **画面 + 声音 + 震动 + 灯带(底座 10×SK6812)**,任一通道都能独立传达"发生了什么"。
4. **宽容物理**:速度封顶、强阻尼、撞墙滑行而非粘住、按孩子实际握持角度做零点校准。
5. **幼儿安全**:亮度压低、暖色、**无全屏快速频闪**(防光敏)、音量适中且软启停防爆音。

## 2. 硬件平台与依赖

### 2.1 主控:M5Stack Core2 初代(v1.0,SKU K010)

详见 `docs/platform/Core2_v1_0.md`。本应用强相关的板级事实(**全部已在该文档核实,勿猜**):

| 用途 | 事实(出处:`docs/platform/Core2_v1_0.md`) |
|---|---|
| SoC | ESP32-D0WDQ6-V3(**经典 ESP32 / LX6,不是 S3**),IDF target = `esp32` |
| 内存 | 16MB flash + 8MB PSRAM,**但经典 ESP32 直映射上限 ≈4MB**,大缓冲按 4MB 算 |
| 屏 | ILI9342C,SPI,**320×240**,MOSI=23/MISO=38/SCK=18/CS=5/DC=15;RST/BL/PWR 走 AXP192 |
| 触摸 | FT6336U,内部 I2C(0x38),INT=39 |
| **IMU** | **MPU6886**(内部 I2C `0x68`)。**本机由 Bottom2 提供**(Core2 背板缺失);仍是 MPU6886、仍在 0x68,**不是** BMI270,**勿照搬 BMI270 代码** |
| 内部 I2C | G21(SDA)/G22(SCL):AXP192 0x34、FT6336U 0x38、BM8563 0x51、**MPU6886 0x68**(IMU 经 Bottom2 挂同总线) |
| 音频 | NS4168(I2S class-D 功放);SPK_EN 在 **AXP192 IO2**;BCLK=12/LRCK=0/DATA=2 |
| **震动马达** | **AXP192 LDO3** —— 本应用把它当一条触觉反馈通道 |
| 电源 | **AXP192(0x34)接管屏/触摸/喇叭/马达供电**;**不初始化 AXP192 → 屏黑、无声、无触摸** |
| 电池 | **由 Bottom2 提供 500mAh**(Core2 自身 390mAh 背板缺失);容量仍有限,**必须做 idle 自动降亮/休眠省电** |

> ⚠️ **强制读一遍 `docs/platform/Core2_v1_0.md` 第 2 节(AXP192)和第 6 节(strapping)。** 音频脚 G0/G2/G12 都是
> strapping,已被板载占用,正常使用别动。

### 2.2 必装底座:M5GO Bottom2

详见 `docs/platform/M5GO_Bottom2.md`。**本应用硬依赖底座**——本机 Core2 缺背部扩展模块,**IMU 和电池都靠它**:

- **IMU(MPU6886 @ 0x68,内部 I2C G21/G22)**:倾斜迷宫的命根子。没有它读不到倾斜,游戏无从谈起。
  **驱动与 Core2 自带 MPU6886 完全一致**(同芯片同地址同总线),只是物理来源是底座,代码无须为底座特化。
- **500mAh 电池 + TP4057 充电**:本机的供电来源。
- **10 颗 SK6812 RGB 灯条(数据线 G25)**:**常驻反馈通道之一**(见 §6、§12),不再是可选氛围。
- 额外引出 **PORT.B(G26 DAC/G36 ADC)** 和 **PORT.C(UART2,G13/G14)**——本应用用不到,但「摇一摇」等其它应用可能用。

> ⚠️ **叠底座后 G25 被灯条占用,不能再当 DAC。** 若 Core2 背板仍在(本机不是这种情况),
> 底座 MPU6886 与背板那颗**同在 0x68 不能共存**,实际生效的是物理接着的那颗——本机背板缺失,
> 生效的就是底座这颗,干净无歧义。

### 2.3 组件依赖(已查 ESP 组件库,见第 16 节审计)

| 依赖 | 用途 | 备注 |
|---|---|---|
| `espressif/m5stack_core_2` (BSP, v3.0.1+) | AXP192 电源 / LCD / 触摸 / **LVGL** / NS4168 喇叭一站式 | target esp32;V1.0 配 `CONFIG_BSP_PMU_AXP192` |
| `espressif/esp_codec_dev` | 喇叭播放(BSP 已间接依赖) | NS4168 经 I2S,SPK_EN 由 BSP/AXP 管 |
| LVGL(随 `esp_lvgl_port`) | 全部 UI 渲染 | BSP 已带 `bsp_display_start*` |
| **自写** `components/imu_mpu6886` | **BSP 不含 IMU**(`BSP_CAPS_IMU=0`),必须自己写 | 复用 BSP 的 I2C 总线句柄,见 §8 |
| `espressif/led_strip` | **底座灯条(必需)**——10×SK6812 @ G25,RMT 后端 | 常驻反馈通道,见 §12 |
| (备选)`jbrilha/esp_lcd_ili9342` | 若 BSP 的 ili9341 兼容初始化出现偏色/偏移再换 | 默认先用 BSP 自带 |

> **关键结论:BSP 把最容易踩的 AXP192 初始化、LCD/触摸/LVGL/喇叭都包好了。
> 你主要要自写的是 ①MPU6886 倾斜驱动 ②游戏逻辑/物理/渲染 ③震动与音效编排。**

## 3. 软件架构

### 3.1 组件划分(遵循 `AGENTS.md` 第 5 节:每个外设独立 component,应用逻辑住 main)

```
project/
├── main/
│   ├── app_main.c              # 初始化 BSP、起任务、进状态机
│   ├── game_state.c/.h         # 状态机(见 §7)
│   ├── physics.c/.h            # 倾斜→物理→碰撞(见 §8)
│   ├── maze.c/.h               # 迷宫数据 + 关卡库 + 求解性校验
│   ├── render.c/.h             # LVGL 绘制(静态迷宫层 + 动态球 + 特效,见 §9)
│   ├── feedback.c/.h           # 反馈编排器:把"事件"分发到 视觉/音频/触觉/灯带
│   └── parent_menu.c/.h        # 家长隐藏菜单(音量/亮度/难度)
├── components/
│   ├── imu_mpu6886/            # MPU6886 最小驱动:输出已校准的倾斜向量(§8.1)
│   ├── haptics/                # 震动马达(AXP192 LDO3)+ 震动模式库(§11)
│   ├── maze_audio/             # 音效播放(esp_codec_dev),内置 PCM 片段(§10)
│   └── ledstrip_fx/            # 底座 SK6812 灯效(§12),常驻通道(本机底座必装)
└── ...
```

> **应用逻辑不直接依赖裸驱动**:physics 只问 `imu_mpu6886` 要"倾斜向量",不碰 I2C 寄存器。

### 3.2 任务划分(FreeRTOS)

| 任务 | 频率 | 职责 |
|---|---|---|
| LVGL task | BSP 自带 | 屏幕刷新(别在别处直接画,**所有 LVGL 调用包 `bsp_display_lock/unlock`**) |
| `game_task` | **60 Hz**(固定 dt) | 读 IMU → 走物理 → 检测碰撞/到达 → 更新球位置(经 LVGL lock)→ 发反馈事件 |
| `audio_task` | 事件驱动 | 从队列收音效请求,串到 esp_codec_dev 播放 |
| `feedback`/`haptics`/`ledstrip` | 事件驱动 / 低频 | 各自消费事件队列,异步执行,**不阻塞 game_task** |

> game_task 用固定 dt(1/60 s)推进物理,渲染跟不上时物理可多步、画面丢帧,但**手感不变**。
> 任务间用 queue 传"事件"(撞墙 / 接近目标 / 到达 / idle 等),解耦反馈通道。

## 4. 游戏机制

### 4.1 迷宫表示

- **网格制**:迷宫 = `W×H` 个格子,每格记录四面墙的开/闭(或用"墙矩形列表")。
- **3~4 岁要极简**:推荐 **6~8 列网格**,**宽走廊**(走廊≈整格宽、球直径约 0.6 格,留足滚动余地,见 §19),路径短(1~3 个拐弯)。
- 一份**手工编排的关卡库**(数据表,**共 4 关、一关一个世界**,见 §19),**不要用纯随机生成**——必须保证可解、宽容、难度可控。
  随机会生成死胡同密布、对幼儿过难的迷宫。
- 关卡库按难度递增(见 §4.5),每关字段:网格墙体、起点、目标点、可选收集物、主题世界、提示等级。配色/形状语言见 §18。
- **每关都要在加载时跑一次可解性校验**(BFS 起点→目标,确认连通);校验失败直接 `ESP_ERROR_CHECK` 式报错,别让幼儿卡死。

### 4.2 坐标与单位

- 物理用**屏幕像素**为单位(320×240 工作区,顶部留 ~24px 状态条可选)。
- 格子像素尺寸 `cell = min(屏宽/W, 屏高/H)`,迷宫居中。
- 球半径 `r ≈ 0.3 × cell`(直径约 0.6 格;本网格下 ≈ 12px / 直径 24px,**具体见 §19,机上可微调**)。

### 4.3 倾斜→物理 模型(详细推导见 §8,这里给设计意图)

经典"滚珠迷宫":倾斜越大→给球的加速度越大;有摩擦阻尼让它会停;速度封顶防失控。
**对幼儿要"重而稳"**:大阻尼、低封顶速度、明显死区,宁可慢一点也别飘。

### 4.4 碰撞(幼儿友好)

- 球 vs 墙:**滑行式**——撞墙只抵消"法向"速度分量,保留"切向"分量,球沿墙面顺滑滑过,
  **不要让球粘在墙上或被卡住**。法向给极小回弹(restitution ≈ 0.2),听感/手感像轻轻"顶"一下。
- 撞墙瞬间发 `EVENT_WALL_BUMP`(带撞击强度 = 法向速度),供反馈编排器决定震动/音效力度。
- 球永远不能穿墙、不能逃出迷宫边界(边界当墙处理)。

### 4.5 难度递进(无失败、无计时)

- **不计分、不计时、无 game over**。唯一"奖励"是到达后的庆祝。
- **共 4 关,一关一个世界,难度逐关递增**(每关都换新风景 → "变难"感觉像"去新地方探险",不是题变难;详见 §19):
  - **第 1 关 · 草地(Tier 0,3 岁起步)**:近乎直线,0~1 个拐弯,目标超大,**显示一条淡淡的面包屑引导**(发光路径)。
  - **第 2 关 · 海边(Tier 1)**:1~2 个拐弯,引导变淡。
  - **第 3 关 · 星空(Tier 2,4 岁)**:2~3 个拐弯,无引导,加 1 颗路途收集星。
  - **第 4 关 · 糖果(奖励关)**:3~4 个拐弯封顶,无引导,1 颗收集星;仍是宽走廊、零失败。
- 到达目标 → 庆祝 → **自动进入下一关**;**走完第 4 关循环回第 1 关**(避免越来越难直到挫败)。
- 可选"收集物":路途中一颗会发光/弹跳的星,滚过即收;**收集是加分项不是必经**(目标点才是过关条件),
  教"顺路经过"的空间概念但不增加失败风险。
- 家长菜单可切"更小/更大"难度档(§13):比如"更小"只跑前两关、引导常驻。

## 5. UI / 视觉设计

> 屏幕 320×240 横向。**一切以"不识字也全懂"为准**:用脸、颜色、动作、图标,不用文字承载信息。

### 5.1 角色化的"球"(灵魂所在)

- 球**不是冷冰冰的圆点,是个有表情的小角色**(一张笑脸 / 小动物 / 小精灵)。
- **眼睛朝运动方向看**(velocity 方向),静止时眨眼;速度快时身体轻微**挤压拉伸(squash & stretch)**。
- 撞墙时**脸挤扁 + 短促晃动**(被"顶"了),配合 §6 的反馈,**表情是搞笑而非难过**(零失败原则)。
- 接近目标时**笑得更开心**,到达时**跳起来欢呼**。

### 5.2 目标点("家")

- 一个清晰、诱人的目标:窝 / 房子 / 星星 / 饼干。**持续柔和脉动(缓慢放大缩小)** 吸引视线。
- **球越靠近,脉动越快、并冒出小亮点**(proximity feedback,建立"快到了"的期待)。

### 5.3 墙体与场景

- 墙**厚实、圆角、友好**(像树篱 / 软垫),高对比但不刺眼;走廊明亮、墙体沉稳。
- **主题随关卡换皮**(草地 / 海洋 / 太空 / 糖果),只换配色和贴图、不换玩法——给重复游玩新鲜感。
- **配色压亮度、偏暖**(见 §13 安全);避免纯白满屏。

### 5.4 各画面状态(对应 §7 状态机)

| 画面 | 内容 |
|---|---|
| **标题 / Attract** | 角色挥手 + 轻快循环背景音 + "倾斜或点屏开始"的**图标动画**(一只手做倾斜手势) |
| **校准 Calibrate** | "把它端平~"的可爱动画(角色站到水平仪中央),0.5~1s 采样握持零点(§8.2) |
| **游戏 Play** | 迷宫 + 球 + 目标;Tier 0/1 叠加**引导虚线**;可选顶部极简状态(电量/家长入口,隐蔽) |
| **撞墙(瞬态)** | 不切画面,仅球挤扁 + 墙体被撞处**短促泛光一下**(单次、非频闪) |
| **庆祝 Win** | 全屏欢乐:**柔和彩纸/星星迸发** + 角色跳舞 + 过关音乐 + 震动;停留 ~2s 自动进下一关 |
| **打盹 Idle** | 长时间无动作 → 角色打哈欠/睡着;轻微"快回来"摇晃 → 重新吸引走神的幼儿 |

### 5.5 动画基调

- 多用**缓动(ease-in-out)、弹性(overshoot)**,让一切有"弹弹的"生命感(幼儿超爱)。
- 庆祝的彩纸/星星要**轻柔飘落**,**不是**高频闪烁(防光敏 + 不刺眼)。

## 6. 交互反馈系统(本应用的重头戏)

**四条反馈通道**(画面 / 音频 / 触觉 / 底座灯带),同一事件**多通道同时触发**(冗余),由 `feedback.c` 编排器统一分发。
反馈强度可由事件携带的"强度"参数缩放(如撞墙力度)。

### 6.1 反馈矩阵(事件 → 各通道)

| 事件 | 视觉(屏) | 音频(NS4168) | 触觉(马达) | 灯带(底座 10×SK6812) |
|---|---|---|---|---|
| **开机** | 角色登场动画 | 轻柔开场音 | 一下短"你好"震动 | 暖色渐亮呼吸 |
| **校准完成** | 角色就位、"开始啦" | 上扬叮咚 | — | 闪一下绿 |
| **滚动中** | 球眼睛朝向 + 挤压拉伸 | (可选)极轻"咕噜"随速度,默认关 | — | (可选)随球位置流光 |
| **撞墙** | 球挤扁 + 撞点泛光(单次) | 柔和"嘭/啵"(力度随撞击) | **短促轻震**(力度随撞击) | 撞侧一下微闪 |
| **接近目标** | 目标脉动加快 + 冒亮点 | 渐升的"快到了"叮铃 | — | 向目标色渐变加亮 |
| **收集到星** | 星被吸入 + 小迸发 | 清脆"叮!" | 极短一下 | 一圈高亮扫过 |
| **到达目标(过关)** | 全屏庆祝 + 角色跳舞 | 欢快过关小调 + "耶!" | **欢庆三连震**(节奏型) | **彩虹迸发/转圈** |
| **打盹 idle** | 角色睡着 | 渐弱至静音 | — | 低亮极慢呼吸 |
| **唤醒** | 角色伸懒腰 | 轻"叮" | 一下轻震 | 渐亮回常态 |

> **强度分级**:撞墙力度分 轻/中/重 三档(按法向速度),分别映射不同音量/震动时长,
> 让"轻轻碰"和"猛地撞"手感不同——这是质感关键。

### 6.2 通道实现要点(细节见各对应章节)

- **视觉**:§9。所有变化经 LVGL,**包 `bsp_display_lock/unlock`**。
- **音频**:§10。**软启停防爆音**:整局保持喇叭使能,靠淡入淡出/播静音,**不要每个音效 toggle SPK_EN**。
- **触觉**:§11。震动是本应用相对独特的一条线,撞墙/过关必给,**短而不腻**。
- **灯带**:§12。底座 10×SK6812 **常驻通道**。设计上**屏幕仍是主信息通道**(孩子视线在屏上,灯带在底座下方),
  灯带做氛围与情绪放大(撞墙微闪 / 接近渐变 / 过关彩虹),**不让灯带成为某事件的唯一线索**——这是为清晰,不是为兼容缺底座。

## 7. 状态机(`game_state.c`)

```
            ┌────────────────────────────────────────────────┐
            ▼                                                │
[BOOT] → [ATTRACT] → [CALIBRATE] → [PLAY] ──(到达目标)──► [WIN] ─┘ (载入下一关→PLAY)
            ▲             ▲           │ │
            │             │           │ └─(撞墙:瞬态,不离开 PLAY,只发事件)
            │             │           │
            │             └───────────┘ (家长菜单返回可触发重新校准)
            │                         │
            └──────[IDLE_SLEEP]◄───────┘ (PLAY/ATTRACT 中长时间无动作)
                       │
                  (检测到动作)→ 唤醒 → 回到来时状态
```

状态说明:

- **BOOT**:`bsp_i2c_init` → BSP 起 LVGL/显示/音频 → IMU 驱动 init(`ESP_ERROR_CHECK` 包住每个 init,见 `AGENTS.md` §3)→ 起任务。失败要**显式日志**,别静默。
- **ATTRACT**:标题画面 + 循环招呼;倾斜超过阈值 **或** 触屏 → CALIBRATE。
- **CALIBRATE**:采样 0.5~1s 重力均值作为"零点"(孩子怎么握就以那个姿态为水平,见 §8.2)→ PLAY。
- **PLAY**:主循环。检测到达 → WIN;长时间无倾斜变化 → IDLE_SLEEP。
- **WIN**:播庆祝(§6),~2s 后载入下一关(同档循环)→ PLAY。
- **IDLE_SLEEP**:降亮度/省电、角色睡着;检测到动作 → 唤醒回原状态。长时间(如数分钟)可进一步深度休眠省电(电池仅 390mAh)。

## 8. IMU 倾斜 → 物理(`imu_mpu6886` + `physics.c`)

### 8.1 MPU6886 最小驱动(`components/imu_mpu6886`)

- **挂在内部 I2C(G21/G22),地址 0x68。复用 BSP 已初始化的 I2C 总线**:
  调用 `bsp_i2c_get_handle()` 拿 `i2c_master_bus_handle_t`,**不要自己再 init 一条 I2C**(会和 AXP192/触摸抢)。
- **本机这颗 MPU6886 物理上来自 Bottom2 底座**(Core2 背板缺失),但仍是 MPU6886、仍在内部 I2C 的 0x68,
  **驱动代码与"Core2 自带 IMU"完全相同**,无须为底座特化——对软件透明。
- 驱动职责很窄:**初始化 + 周期读三轴加速度(g)**。陀螺仪本应用可不用(纯重力倾斜即可),用了更稳也行。
- **务必按 MPU6886 datasheet 配寄存器(WHO_AM_I / PWR_MGMT / ACCEL_CONFIG / 数据寄存器布局)。
  绝对不要照搬 BMI270 的寄存器**(见 `docs/platform/Core2_v1_0.md` §7 警示)。开发时**先查 MPU6886 datasheet 再写**,
  寄存器地址/期望 ID 逐一核对,不要凭印象填(`AGENTS.md` §1)。
- 量程取 ±2g 足够(只测重力倾斜),输出标准化为 g。

### 8.2 校准(对幼儿至关重要)

孩子端着的姿态**不是水平**。进 PLAY 前采样 0.5~1s 加速度均值 `(a0x,a0y,a0z)` 作为**中性零点**:
之后的"倾斜"= 当前重力相对该零点的偏移。这样**无论孩子怎么舒服地端着,那个姿态就是"平"**,极大降低门槛。
家长菜单或每次进关可重新校准。

### 8.3 倾斜 → 加速度 → 速度 → 位置

每帧(固定 `dt = 1/60`):

1. 读 `(ax,ay,az)`(g)。
2. **低通滤波**去抖:`a_filt = a_filt*(1-α) + a_raw*α`,`α ≈ 0.25`。
3. 取**屏幕平面内分量**相对零点的偏移:`tilt = (a_filt.xy - a0.xy)`。
   - **屏幕轴 ↔ IMU 轴的映射(符号/交换)要实测确定**:`docs/platform/Core2_v1_0.md` §7 明确 MPU6886 轴向
     **不可照搬 BMI270**。提供 `axis_swap_xy` / `invert_x` / `invert_y` 配置项,**首版用串口打印实测标定**。
4. **死区**:`|tilt| < DEADZONE` → 视为 0(防止"平放"时球自己飘)。
5. `accel = tilt * GAIN`(GAIN 可调,起步偏小)。
6. `vel += accel * dt;  vel *= DAMPING`(`DAMPING ≈ 0.9`/帧,强摩擦让球会停)。
7. **封顶**:`|vel| = min(|vel|, V_MAX)`(幼儿一定会过度倾斜,**必须封顶**)。
8. `pos += vel * dt`。
9. **碰撞解算**(§4.4):逐墙检测,法向推出 + 抵消法向速度(切向保留=滑行),撞击发 `EVENT_WALL_BUMP(强度)`。
10. **到达检测**:`dist(pos, goal) < goal_r` → `EVENT_REACH_GOAL` → 状态机转 WIN。
11. **接近检测**:`dist(pos, goal)` 进入若干阈值环 → `EVENT_NEAR_GOAL(等级)`。

可调参数集中放一个 `tuning.h`(`GAIN / DAMPING / V_MAX / DEADZONE / α / restitution / goal_r` 等),
**现场抱着真机调到"重而稳、好上手"**——这些数值的手感只能 build+flash+实测,别指望一次写对。

> ✅ **已定案(2026-06-30 实机标定,用户确认手感 OK)**:轴映射 `TILT_INVERT_X=1, TILT_INVERT_Y=0, TILT_SWAP_XY=0`
> (实测上下左右都反,X/Y 同时取反);手感值 `GAIN=1800 / DAMPING=0.90 / V_MAX=220 / DEADZONE=0.06 / α=0.25 / restitution=0.2`。
> 全部在 `main/tuning.h`,可继续微调。

> **PSRAM/IRAM 注意**(`AGENTS.md` §6 通用坑,虽然本板是经典 ESP32):本应用不用 BLE,
> 主要风险是大缓冲。LVGL 绘制缓冲走 BSP 默认(部分缓冲、DMA);整张迷宫静态层若用 canvas
> 放 PSRAM(150KB 量级,落在直映射 4MB 内 OK)。**IMU 读取在 I2C 任务上下文,无 DMA 同步对象落 PSRAM 之虞。**

## 9. 渲染方案与渲染预算(`render.c`,基于 LVGL)

> **Confirmed via espressif-docs / esp-component-registry**:Core2 BSP 提供 `bsp_display_start*`
> 起好 LVGL(`esp_lvgl_port`),底层 esp_lcd SPI panel 用 `esp_lcd_panel_draw_bitmap` 局部刷新。
> 移动的球只更新脏矩形,无需整屏重绘。

> ⚠️ **这一节是硬规矩,不是建议。** Core2 是经典 ESP32(LX6),**没有任何 2D 加速 / DMA2D / PPA**,
> 所有合成都是 CPU 软件渲染;屏走 **SPI**。**渲染瓶颈是 SPI 带宽,不是算力。** 实现的人若一上来就
> "整屏刷新",帧率必死。下面的分层 + 预算就是为了避免这件事——美术做得丰富没问题,关键是
> **"每帧到底有多少像素在动"**。

### 9.1 三层渲染模型(成败核心)

| 层 | 何时画 | 代价 | 内容 |
|---|---|---|---|
| **静态层** | **每关载入时画一次**,关内不再重画 | 付一次(~31ms,可接受) | 主题背景、墙体/树篱、装饰花草、烘好的柔光/阴影、引导面包屑 |
| **动态层** | 每帧,**只刷脏矩形** | 极小(球 ~2ms) | 球(`lv_image`/小精灵,每帧改坐标)、家的脉动、收集物 |
| **特效层** | 事件触发,短生命周期 | 视情况,见 §9.5 | 撞墙泛光、接近亮点、庆祝彩纸 |

> **关键推论:静态层再"丰富"几乎是免费的(只付一次)。** 想让画面好看,把东西画进静态层,
> 别让它每帧动。LVGL 的 dirty-rect 天生这么干:背景是一张大图对象,只移动球对象,它只失效球经过的小块。

### 9.2 帧预算:瓶颈是 SPI 带宽

按 BSP 默认像素时钟 **40MHz**、RGB565(16bpp)估算:

- 吞吐 ≈ 40e6 / 16 ≈ **2.5M 像素/秒**(≈ 2.5 px/µs)。
- **整屏** 320×240 = 76,800 px → 推一帧 ≈ **31ms** → **整屏重绘天花板仅 ~30fps**(还没算渲染合成)。
- **球脏矩形** ~70×70 ≈ 4,900 px → ≈ **2ms** → 球可稳 **60fps**,只占带宽 ~12%。

**硬规矩:**

1. **永不每帧整屏重绘。** 整屏级刷新只允许出现在两处:**进关载入** 和 **庆祝**。
2. 静态层进关画一次(可放 **PSRAM canvas**,150KB,在直映射 4MB 内);关内只重画动态/特效的脏矩形。
3. **每帧"在动的像素"总预算**:常态 game loop **≤ ~15,000 px/帧**(球 + 家脉动 + 少量氛围点,换算 ≤ ~6ms),
   给球留足 30–60fps。超了就砍氛围动效,别砍球的流畅。
4. **LVGL 用部分缓冲**(BSP 默认 `BSP_LCD_DRAW_BUFF_SIZE = H_RES*50`),**别开 `full_refresh`、别分配整屏 framebuffer 做双缓冲**。
5. SPI flush 走 **DMA 异步**(esp_lcd/BSP 已是),刷屏时 CPU 不空等;`flush_ready` 在传输完成回调里给。
6. 所有 LVGL 操作包 `bsp_display_lock(0) … bsp_display_unlock()`(BSP 要求)。

### 9.3 色彩:扁平优先,RGB565 注意 banding

- 屏是 **RGB565(R5 / G6 / B5)**,绿才 6 位。**软渐变会肉眼可见地分层(banding)**,而且要么每帧算、要么占空间。
- **扁平大色块优先**——既最省,也正好是更对的幼儿"chunky"审美(样图就是这么做的)。
- 真要渐变 / 柔光 / 纹理 → **烘进 flash 里的预渲染图**,载入时贴一次,**不每帧算**。需要可在烘图时加 dithering 缓解 banding。

### 9.4 alpha / 发光 / 脉动:烘进静态层,别每帧混合

- 经典 ESP32 上 **alpha blend 是纯 CPU,很贵**。半透明发光、阴影 → **预渲染进背景**(付一次),不要每帧全屏 alpha。
- **脉动 / 呼吸** → 缩放一个**不透明**小精灵,或在 2–3 张预渲染状态图间切换,**只动那一小块**。
- **引导面包屑"逐渐变淡"** → 用几级**预渲染好透明度的图**切换,而不是每帧重算 alpha。
  (此处修正 §5 / 早期描述里"半透明路径逐渐变淡"的实现方式:落地必须是分级预渲染,不是每帧 alpha。)

### 9.5 各动效的帧率档位

| 动效 | 帧率 | 约束 |
|---|---|---|
| 物理步进 | **60Hz** | 纯数学 + 读 IMU,便宜,与渲染解耦 |
| 球渲染 | 跟 **60fps** | 小脏矩形(~2ms),第一优先,永不为别的让路 |
| 家脉动 / 收集物 | 球的帧率 | 小区域不透明缩放 |
| **氛围微动**(星眨、窄浪带) | **5–10fps** | **各自小区域,合计动的像素压在 §9.2 预算内;绝不是整面墙在动** |
| **庆祝彩纸** | 可掉到 **15–20fps** | 此刻球不归孩子控、察觉不到;**限数量 / 限面积**,或做成**预渲染精灵序列**,别上百粒子每帧重算;柔和、非频闪 |

> 与 §6 反馈矩阵对齐:那里写的"庆祝全屏"、"接近冒亮点"、"流光"等,**全部按本节档位实现**——
> 局部小精灵 + 低帧率,而不是整面动。

### 9.6 ili9342 偏色 / 偏移备选

> 若 BSP 的 ili9341 兼容驱动在本屏出现**偏色 / 左右偏移 / 镜像**,改用 `jbrilha/esp_lcd_ili9342`
> 或给 panel 设 `mirror / swap_xy / gap`——这是 ILI9342C vs ILI9341 常见差异点,开发时实测确认。

> **亮度**:`bsp_display_brightness_set(...)` 压到舒适档(见 §13);idle 时进一步调低(也省电)。

## 10. 音频方案(`maze_audio`,基于 esp_codec_dev)

- **走 BSP 音频**:`bsp_audio_codec_speaker_init()` 拿 codec handle,`esp_codec_dev_open/write/close`
  播放 **PCM**。BSP 把 NS4168 经 I2S 封装好,SPK_EN(AXP192 IO2)由 BSP/AXP 管。
- 音效素材:**预先准备好的小 PCM 片段**(16kHz/16-bit/mono 即可),放 SPIFFS 或编进 flash 数组。
  事件 → 选片段 → 投入 `audio_task` 队列播放。
- **防爆音(对应 EchoS3R 同类经验,见 `AGENTS.md` 音频纪律)**:
  - **整局保持喇叭 open**,空闲时写静音,**不要每个音效 close+open / 反复 toggle SPK_EN**(会咔哒)。
  - 片段首尾做**短淡入淡出**。
  - **只在采样率/声道变化时才 reopen**;统一一种采样率最省事。
- **音量**:`esp_codec_dev_set_out_vol(...)`,默认中等(幼儿+家庭环境),家长菜单可调,留上限防过响。
- 背景音乐(Attract/Win)可循环;游戏中默认不放持续 BGM,以免盖过事件音效(可家长开关)。

> **滚动声默认关**:持续滚动音容易腻/吵,首版只做"事件音"(撞墙/接近/收集/过关)。若要加,做成可调极轻的随速音。

## 11. 触觉:震动马达(`haptics`,AXP192 LDO3)

- 马达由 **AXP192 LDO3** 供电(见 `docs/platform/Core2_v1_0.md` §2)。**优先用 BSP 的
  `bsp_feature_enable(BSP_FEATURE_VIBRATION, true/false)`**(BSP 暴露了该 feature);
  若需更细的时长/节奏控制,再经 AXP192 直接控 LDO3 通断——**那部分先查 AXP192 寄存器再写,勿猜**。
- **震动模式库**(短而不腻,幼儿友好):
  - `BUMP_LIGHT/MED/HARD`:撞墙,约 30/60/100ms 单次,力度随撞击。
  - `COLLECT`:收集星,极短一下(~25ms)。
  - `WIN`:过关三连(如 80ms ×3,间隔 80ms),欢庆节奏。
  - `HELLO/WAKE`:开机/唤醒,一下轻震。
- 震动在独立任务/定时器里跑,**绝不阻塞 game_task**。
- 提供**总开关**(家长菜单)——有的家庭夜间不想要震动。

## 12. 底座灯带(`ledstrip_fx`,SK6812 ×10 @ G25)—— 常驻反馈通道

> 底座必装,灯带**常驻启用**。屏幕仍是主信息通道(§6.2),灯带负责氛围与情绪放大。

- 10 颗 SK6812 单数据线 **G25**,WS2812 时序、GRB,用 `espressif/led_strip` + RMT 后端,灯数=10
  (见 `docs/platform/M5GO_Bottom2.md` §2)。
- 效果:开机暖色呼吸 / 游戏中随球位置或主题色微光 / 接近目标向目标色加亮 / **过关彩虹迸发或转圈** / idle 极慢呼吸。
- **亮度压低**(`docs/platform/M5GO_Bottom2.md` 明确 SK6812 满亮度刺眼且耗流;幼儿常态务必降亮)。
- ⚠️ G25 本是 Core2 的 DAC,叠底座后被灯条占用,**别再把 G25 当别的用**。
- RMT 通道别和别处冲突;灯带刷新放低频任务,**不阻塞 game_task**。
- 🔴 **必须自己打开 M-Bus 5V(AXP192 EXTEN),否则灯带全黑!**(2026-07-01 实测踩中,耗时排查)
  灯带 VCC 吃 M-Bus 5V,这条 5V 由 Core2 的 **SY7088 升压**提供,升压使能挂在
  **AXP192 EXTEN(REG 0x12 bit6)**。**Espressif BSP 只配屏/触摸/喇叭/震动那几路 rail,从不开 EXTEN**
  → 底座灯带没供电、**完全不亮**。而数据线 G25 是 3.3V 逻辑照常翻转、`led_strip_refresh` 照样返回
  `ESP_OK`,**极易误判成"灯带坏了/驱动错了"**(实则烧别的软件能亮就是因为它们开了 EXTEN)。
  修法见 `main/power.c`:复刻 M5Core2 `SetBusPowerMode(0)`——GPIO0 设 LDOio 3.3V 输出 + 置 EXTEN。
  **须在 `bsp_display_start` 之后调用**(BSP 的 AXP192 初始化会把 REG 0x12 清 0)。深度省电时 `power_bus_5v(false)` 关掉可省电。

## 13. 幼儿安全 / 无障碍 / 家长控制

**安全(硬性)**:
- **亮度压低 + 暖色**,避免满屏纯白长时间直射眼睛;idle 进一步调暗。
- **无全屏快速频闪 / 无高频强对比闪烁**(光敏安全);庆祝特效柔和飘落,撞墙泛光为**单次**非连闪。
- **音量有上限**,默认适中,软启停防爆音突响吓到孩子。

**无障碍(针对 3~4 岁)**:
- **不识字**:信息全靠 脸/颜色/动作/图标/声音;文字仅装饰。
- **大目标、大球、宽走廊、慢球、强阻尼、按握持角校准** —— 全方位降门槛。
- **撞墙滑行不卡死**,永不失败。

**家长控制(隐藏,防误触)**:
- 入口:**底部三个触摸虚拟按键之一长按 3s**(幼儿不会无意触发),或开机时特定手势。
- 菜单项:**音量、屏幕亮度、震动开关、难度档(更小/更大)、背景音乐开关、重新校准**。
- 菜单用大图标 + 滑块,家长一眼可用;退出回游戏并重新校准。

## 14. 电源 / 健壮性

- **AXP192 必须先配好**(BSP `CONFIG_BSP_PMU_AXP192` 负责)——否则屏黑/无声/无触摸,**别误判成代码或接线坏**
  (`docs/platform/Core2_v1_0.md` §2 反复强调)。
- **电池为 Bottom2 的 500mAh**(本机供电来源):idle 自动降亮 → 更长 idle 进低功耗;可在 Attract/idle 关喇叭、降 LVGL 刷新、灯带降到极低。
  充电经底座 TP4057(pogo 磁吸)或 Core2 USB-C。
- 设备会被**摇晃/磕碰/猛倾**:IMU 读数滤波 + 速度封顶 + 死区,避免异常值让球瞬移或卡墙。
- 每个驱动 init 用 `ESP_ERROR_CHECK` 包住,**静默 init 失败是第一耗时坑**(`AGENTS.md` §3)。
- 改 `sdkconfig.defaults` 后**必须删 `sdkconfig` 再 fullclean build**(`AGENTS.md` §3)。

## 17. 坑位速查(开工前再扫一眼)

- **AXP192 不初始化 = 屏黑/无声/无触摸**,别误判硬件坏(BSP 已处理,但 Kconfig 选错 PMU 版本会中招)。
- **IMU 是 MPU6886 不是 BMI270**,寄存器与轴向**都别照搬 BMI270**;轴↔屏映射实测标定。
- **音频脚 G0/G2/G12 是 strapping**、已被板载占用,正常别动;喇叭别反复 toggle SPK_EN(咔哒)。
- **经典 ESP32 PSRAM 直映射 ≤4MB**,大缓冲按 4MB 算。
- **电池为 Bottom2 500mAh**(Core2 背板缺失),务必做 idle 省电。
- **改 sdkconfig.defaults 后删 sdkconfig 再 fullclean**。
- **底座必装**:IMU(MPU6886 @ 0x68)与电池都来自 Bottom2,**不接底座就没倾斜、没电——游戏跑不了**。
  开机/M0 先确认底座在位、I2C 扫得到 0x68;灯带 G25 常驻启用,G25 不能再做别的。
- 🔴 **灯带全黑先查 EXTEN**:底座灯带吃 M-Bus 5V(SY7088 升压,AXP192 EXTEN 使能),**BSP 从不开 EXTEN**;
  数据线正常翻转、`refresh` 返回 OK 也可能全黑。必须自己开 EXTEN(见 `main/power.c`、§12)。**别误判成硬件坏**。
- 🔴 **`brightness_set(0)` 不熄屏**:Core2 背光=AXP192 **DCDC3**,BSP 的亮度只调 DCDC3 电压(0% 仍 ~2.95V,还有微光)。
  深度省电要**真黑屏必须断 DCDC3 使能**(REG 0x12 bit1,见 `power_backlight()`)。与 EXTEN 同类:BSP 不管、得直接控 AXP192。详见 §20.6。
- **打盹判据只看机身动作(IMU),别看球速**:球在残余倾斜下会永远慢爬,用球速判定会**永不打盹**;唤醒要连续多帧去抖,防单帧噪声误唤醒。详见 §20.6。
- **渲染红线:永不每帧整屏重绘**(经典 ESP32 无 2D 加速,SPI 40MHz 整屏 ~31ms → 整屏刷必死帧率)。
  静态层进关画一次、关内只刷脏矩形;扁平色优先(RGB565 渐变会 banding);发光/脉动烘进静态层别每帧 alpha。详见 §9。
- **关卡用手工编排 + 加载时可解性校验**,别用纯随机。

### 18.1 总原则

- **一只吉祥物,四个世界,永远是"回家"**:目标概念恒定(降认知负担),只换风景(新鲜感)。
- **收集物永远是同一颗星 ✦**(孩子学会"星=去捡"一条规则,各世界不换)。
- **大块扁平圆润形 + 暖色 + 压亮度**(chunky 幼儿审美,也最省;少用渐变,要渐变就烘进 flash 图,见 §9.3/§9.4)。

### 18.3 形状语言(跨世界一致)

- **墙**:圆角块(`rx ≈ 0.3×cell`),**顶缘叠 2~3 个浅色椭圆**做"蓬松/起伏"轮廓(树篱叶团 / 浪头 / 云朵 / 糖块),表面撒深色 speckle + 几颗点缀。全部**画进静态层**。
- **路面**:纯扁平底色;可在静态图里烘一层**极淡**纹理点(草点/沙点/星点/饼干点),不每帧画。
- **家**:始终"窝/家"概念,圆/半圆基座 + 暖色;**柔光环预渲染进静态层**(不每帧 alpha,见 §9.4);脉动 = **缩放整张家精灵**(小区域)。
- **星(收集物)**:五角星双色,所有世界统一;弹跳/自转用小精灵。
- **面包屑**:一串小圆点(各世界换色/形);**淡出 = 几级预渲染透明度切换**(不每帧算 alpha,见 §9.4)。

### 18.4 吉祥物「圆圆」

- **身体**:圆形,直径 ≈ `0.6×cell`(本网格 ≈ 24px,`r≈12px`)。
- **脸**:两只白眼 + 深瞳(**瞳孔朝速度方向偏移 ~2px** = "看着要去的方向"),小喙(橙),两点腮红;静止时眨眼。
- **挤压拉伸**:速度大时纵向压扁/横向拉;**撞墙瞬间挤扁一下**(搞笑、非难过,呼应零失败)。
- **各世界换帽**(同一张脸,只换帽贴 → 一张脸精灵 + 4 顶帽):草帽 / 游泳圈 / 星星帽 / 糖果帽。
- **表情态**(预渲染几张状态精灵切换,**不每帧重画脸**):普通 / 开心(接近家)/ 欢呼(到家跳)/ 睡(idle)。

全部是**局部小精灵、5~10fps、各自小区域**,合计"动的像素"压在 §9.2 预算内 —— **绝不整面墙/整片水在动**:

| 世界 | 微动(各自小区域) |

| ① 草地 | 2~3 朵花轻摇(各 ~12px);偶尔一只蝴蝶走直线飞过(小精灵) |

### 20.6 休眠 / 唤醒 / 背光机制(2026-07-01 实机调通)

> 三级状态由 **IMU 单一信号**驱动;灯带与背光是**两条独立 AXP192 电源**,分别断/恢复。踩过的三个坑见本节末——都属「BSP 不管、得直接控 AXP192」那一类。代码在 `main/game_state.c` + `main/power.c`。

**状态流**:`PLAY ──(机身静止 12s)──► IDLE 打盹 ──(再静止 60s)──► DEEP_IDLE 深度省电`;任一休眠态检测到「真的动了」→ `wake_from_idle()` → 回 PLAY。

**核心信号——机身动作量**(`game_task` 每帧算一次):
`s_motion = |Δax| + |Δay| + |Δaz|`(帧间三轴加速度变化,g)。平放静止 ≈0.005~0.03(噪声尖峰偶达 ~0.08),被拿起/倾斜 >0.12(`IDLE_WAKE_THRESH`)。**「有没有人在玩」只由此判定**,与球的运动无关。

**① 进入打盹(PLAY→IDLE)——只看机身动作,不看球速**:
```
s_motion > IDLE_WAKE_THRESH ? still_frames=0 : still_frames++;
still_frames > 750帧(12s) → enter_idle();
```
`enter_idle`:背光 60%→10%(`IDLE_BRIGHTNESS`)、灯带→慢呼吸(`LED_BASE_IDLE`)。

**② 进入深度省电(IDLE→DEEP_IDLE)——计时不被单帧噪声打断**:
```
woke_up() ? wake : (frame++, frame>3750帧(60s) → enter_deep_idle());
```
`enter_deep_idle` 依次:`brightness_set(0)`(先降 DCDC3 电压)→ **`power_backlight(false)`(断 DCDC3 → 背光真全黑)** → 灯带 `LED_BASE_OFF` → `power_bus_5v(false)`(切 M-Bus 5V,断灯带供电 + 省 SY7088 静态电流)→ 轮询周期降到 `DEEP_IDLE_POLL_MS=120`(少唤醒 CPU 省电)。

**③ 唤醒(去抖)**:`woke_up()` 要**连续 `WAKE_DEBOUNCE_FRAMES=3` 帧** `s_motion>0.12` 才算真动(单帧噪声忽略)。`wake_from_idle` 依次:`power_bus_5v(true)` → **`power_backlight(true)`(重启 DCDC3)** → 背光回 `s_play_bright`(60%)→ 灯带 `LED_BASE_AMBIENT` → `HAPTIC_WAKE`(一下轻震)→ 回 PLAY。

**两条独立电源**(均在 `main/power.c` 经 AXP192 读改写):

| 通道 | 供电 | 使能位 | 熄灭方法 |
|---|---|---|---|
| **屏背光** | AXP192 **DCDC3**(电压在 REG `0x27`) | REG `0x12` **bit1**(0x02) | `power_backlight(false)` 清 bit1 |
| **底座灯带** | M-Bus 5V(SY7088 升压) | AXP192 **EXTEN** = REG `0x12` **bit6**(0x40) | `power_bus_5v(false)` 清 bit6 |

**三个坑(为何这么设计)**:
1. **打盹判据不能用球速**:机身平放但相对校准零点有残余倾斜(>死区 0.06g)时,球会**永远以 ~20px/s 慢爬**,`sp>8` 把打盹计数一直清零 → 永不打盹。→ 只看 `s_motion`(机身没动=无人玩,球爬不爬无所谓)。
2. **唤醒必须去抖**:平放时 IMU **单帧**噪声尖峰偶尔 >0.12,会误唤醒并把 60s 深度省电计时**整段作废** → 熬不满。→ 连续 3 帧才算真动。
3. **`brightness_set(0)` 根本不熄屏**:BSP 亮度只调 DCDC3 **电压**(公式 `reg=90+8*pct/100`,范围仅 90~98 ≈ **2.95~3.15V**),0% 还剩 2.95V,背光照亮。→ 真熄屏必须**断 DCDC3 使能**(REG 0x12 bit1)。

> **关键约束复述(勿忘)**:IMU 是 **MPU6886 非 BMI270**;**永不每帧整屏重绘**(§9);关卡手工编排 + 加载 BFS 校验(§4.1/§19)。
> 改 `sdkconfig.defaults` 后删 `sdkconfig` 再 fullclean(§14)。激活环境用 `activate_idf_v6.0.1.sh`。
