## 0. 为什么重写(诊断)

现状问题不是"不够好玩",是**压根没有当游戏设计过**:左边一个表盘、右边一个光点方框、底下一行
数字——这是给开发者自己排障用的仪表台,没有主题、没有角色、没有情绪反馈,根 `CLAUDE.md` §2
的五条设计第一性原则(零失败/即时因果/多通道冗余/宽容物理/幼儿安全)基本没有被认真对待过。
但它的技术底子是好的:`chain_bus` 传输层 + 两个节点驱动已经跑通(`chain_lab.c` 现有
`scan_bus()` 能同时认领 encoder 与 joystick 两个链上节点、`game_task` 在同一个 20Hz 循环里
并发轮询两者),协议层完全不需要动,只缺一层"游戏"。

## 1. 保留不动(红线,严禁重写)

- `components/units/chain_bus`、`components/units/unit_chain_encoder`、
  `components/units/unit_chain_joystick` **协议/传输层零改动**。
- `chain_lab.c` 现有以下函数原样保留,作为游戏层的输入来源,不重写内部实现:
  - `scan_bus()` —— 开机扫链位 1..`CHAIN_MAX_ID`,自动认领 encoder/joystick,支持热插拔重扫。
  - `poll_enc(bool draw)` / `poll_joy(bool draw)` —— 读值 + 边沿检测 + **返回 `activity`
    布尔值**(供 `core2_sleep_kick` 用,已实现,直接复用)。
  - `node_rgb()` —— 节点 RGB 写入节流(值不变不重发,省事务)。
  - `hue2rgb()` —— 色相环转 RGB,派对彩虹沿用。
  - `joy_calibrate_center()` —— 摇杆上电居中校准。
- `game_task` 的 20Hz 轮询节奏、`core2_sleep` 集成、深度省电断 5V→节点复位→唤醒重扫的处理
  (`s_prev_stage == CORE2_SLEEP_DEEP` 分支)**原样保留**。
- 供电/接线/省电坑同 `chain_bus` 头注释与 README 不变(PORT.C 5V = EXTEN)。

**重写(游戏层)**:`ui_create/ui_enc/ui_joy/ui_status` 这套仪表台 UI 折叠为隐藏诊断视图
(§7);新增抓娃娃机的状态机、精灵、战利品系统(§3~§6)。

## 2. 输入 → 机制映射

| 输入 | 复用的现有读数 | 抓娃娃机机制 |
|---|---|---|
| 摇杆 X(`s_joy_nx`) | `poll_joy` 已算好的归一化 X | 吊臂左右移动(screen-space,限制在战利品排范围内) |
| 摇杆 Y(`s_joy_ny`) | 同上 | **v1 不用**(见 §11 开放风险 ①,避免同时要求"两轴定位+拧旋钮下降"三种精细动作叠加,先把一维版本做扎实) |
| 摇杆按键(`s_joy_btn`) | `poll_joy` 已有的 `press_edge` | 备用彩蛋:摇一摇爪子(小抖动+轻响,不影响进度) |
| 编码器计数增量(`s_enc_value` 帧间 delta) | `poll_enc` 已读到的绝对计数,应用侧自己取相邻两帧差值当"转了几格" | 顺时针转 = 爪子下降(累加深度,封顶);逆时针转 = 爪子回收(减深度,下限 0);**中途反悔零成本**,不是必须转到底 |
| 编码器按键(`s_enc_btn`) | `poll_enc` 已有的 `press_edge` | **抓取**:在当前深度触发爪子闭合判定 |

## 3. 状态机

```
[SCAN] ──(scan_bus 认领到至少一个节点)──► [PLAY_IDLE](爪子收起,吊臂可自由横移)
   │ 没认到节点 → 现状"无字提示卡"逻辑不变,周期重扫

[PLAY_IDLE] ──(编码器顺时针转)──► [DESCENDING](爪子随深度下降)
[DESCENDING] ──(编码器逆时针转 / 抓取键提前按下)──► [PLAY_IDLE](中途放弃,零成本零惩罚)
[DESCENDING] ──(深度到底 或 抓取键按下)──► [GRABBING](爪子闭合动画,~400~600ms)
[GRABBING] ──► [ASCENDING](自动上升,~600~900ms,携带战利品或空爪)
[ASCENDING] ──(到达顶部)──► [DEPOSIT](~500ms:战利品落入展示架,或空爪"诶?"弹一下)
[DEPOSIT] ──► [PLAY_IDLE]

跨状态:展示架集满(当前批次战利品类型收齐)──► [PARTY](~3~4s)──► 展示架清空、换新一批战利品 → [PLAY_IDLE]

隐藏分支:诊断手势触发(§7)──► [DIAG_VIEW](现有表盘/光点 UI)──► 再触发一次 → 回 [PLAY_IDLE]
```

- **按键防抖/防重入**:`GRABBING`/`ASCENDING`/`DEPOSIT` 期间再按抓取键无效(状态门控吃掉,
  仿现有 `press_edge` 写法,不排队不触发二次动画)。
- **摇杆在下降/抓取过程中仍可移动吊臂**(不锁定横向自由度),只有爪子的深度受编码器控制,
  两个输入通道保持独立,孩子随时能重新瞄准。

## 4. 反馈矩阵(六类事件 × 六条通道,**节点 RGB 是本卡带独有的第六通道**)

| 事件 | 屏幕 | 音频 | 触觉 | 底座灯带 | 编码器节点 RGB | 摇杆节点 RGB |
|---|---|---|---|---|---|---|
| 吊臂移动 | 小 dirty-rect 位移(同现状光点更新量级) | 静音(连续移动不配音,防刷屏噪音) | — | — | — | 沿用现状既有映射(`nx/ny → r/b`,`chain_lab.c` 已实现,原样保留) |
| 下降(编码器转) | 爪子/缆绳精灵向下延伸 | 每格一声极轻"咔"(仿 tick 类短音,~30ms) | — | 暖色微调暗一档 | **颜色随深度加深**(浅→深,离散几档随深度切换,不做逐帧渐变) | — |
| 抓取(按键) | 爪子闭合动画 | `SND_COLLECT` 风格短音 | `HAPTIC_BUMP_MED`(机械"咔嗒"感) | `LED_FX_FLASH` | 闪白一下(复用现状 `node_rgb(...,255,255,255,...)` 按下逻辑) | — |
| 抓中(上升途中判定成功) | 战利品随爪子一起上升 | 上行音序 | `HAPTIC_COLLECT` | `LED_FX_COLLECT` 扫圈 | 上升期间保持"成功色"(绿) | — |
| 抓空(上升为空) | 爪子空手到顶,轻轻弹一下(**不是失败动画**,定性为"诶?再试试") | 轻"啵"(`HAPTIC_BUMP_LIGHT` 同级,**不用负面/低沉音**) | `HAPTIC_BUMP_LIGHT` | `LED_FX_BUMP` | 中性琥珀色轻闪(**不用红色**,零失败原则) | — |
| 展示架集满 → 派对 | 展示架战利品一起小跳 + 彩纸 | `SND_WIN` | `HAPTIC_WIN` | `LED_FX_WIN` 彩虹 | **两节点同步跑彩虹**(复用现有 `hue2rgb`) | 同左 |

## 5. 零失败处理

- **抓空零惩罚**:不掉血、不计失败次数、不需要"再试一次"提示——直接允许立刻再来。
- **中途反悔零成本**:`DESCENDING` 期间随时能逆时针转回去,不损失进度、不重置吊臂位置。
- **对齐容差要给够**:判定"爪子是否对准战利品"用**中心点容差带**(不要求像素级精确对齐),
  孩子的摇杆微调精度有限,容差带要覆盖典型手部抖动幅度(具体像素值待实机标定)。
- **按键连点无副作用**:`GRABBING`/`ASCENDING`/`DEPOSIT` 期间的额外按键被状态门控静默吃掉,
  不排队、不打断当前动画、不产生"抓了两次"的错觉。
- **深度"转够了自动到底"**:编码器不要求孩子精确数格子,接近最大深度时可以有一点吸附/封顶
  容忍(具体阈值待实机标定,见 §11 风险②)。

## 6. 电源 / 省电集成

- **kick 判据完全复用现状**:`poll_enc`/`poll_joy` 已经在算 `activity`(转动/偏移/按住任一即
  真),游戏层不需要重新发明判据,直接吃现有返回值喂 `core2_sleep_kick`。
- **深度省电唤醒后**:沿用现有 `s_prev_stage == CORE2_SLEEP_DEEP` 分支的重扫逻辑,游戏层在
  此基础上追加"爪子/吊臂复位到安全位置"(爪子收起、吊臂回中),仿 peekaboo/feed_monster 在
  `unit_attach` 时重摆场景的先例,避免节点断电复位后游戏状态与物理输入(编码器计数清零)对不上。

## 7. 诊断模式保留方案(不能丢失排障能力)

`README.md` 明确"Core2 直连 Chain host 未经官方背书",`chain_bus_sniff()` 抓原始字节的排障
路径是这张卡带的额外价值,重写为游戏后不能丢。两个方案(**留给实现时二选一,不是本轮必须拍板
的设计决策**):

- **方案 A(运行时切换)**:编码器按键 + 摇杆按键**同时长按 ~1.5s** → 切换到现有
  `ui_enc`/`ui_joy`/`ui_status` 表盘/光点视图(代码原样保留,只是套一层显示开关);同手势
  再触发一次切回游戏。优点:不用重新编译就能现场排障。缺点:双键长按容易和正常抓娃娃机操作
  (孩子乱按)误触发,且平台已知"长按手势不易触发/易被误触"的问题(根 `CLAUDE.md` §8 已记录
  tilt_maze 家长菜单长按同款坑),需要实机验证长按判定的可靠性与误触率。
- **方案 B(编译期开关)**:`tuning.h` 里 `#define CHAIN_LAB_DIAG_MODE 0/1`,置 1 时整个
  app 编译成现状的诊断台(游戏层代码不参与),置 0 才是游戏。优点:零运行时复杂度、零误触
  风险。缺点:排障时需要改宏重新烧一次,没有"现场切换"的便利。

## 10. 渲染预算合规(根 `CLAUDE.md` §6)

- 静态层:战利品排背景、展示架框、吊臂导轨——场景载入时画一次。
- 动态层:吊臂横移(~30×20px 小方块)+ 爪子/缆绳纵向延伸(细长矩形,随深度改高度,同
  busy_knobs 柱子长高的既有手法,零 alpha 成本)——与现状表盘指针(~9px 半径圆点)/光点更新
  同一量级,已被现有 chain_lab 证明够便宜。
- 战利品精灵:~30×30px 静态块,被抓中时随爪子一起移动(单个小移动精灵,同吊臂本体量级)。
- 派对:展示架战利品小跳 + 彩纸,走 §6.5 庆祝档(15~20fps 可接受),战利品种类数按 §9
  `PRIZE_TYPES=5` 封顶,避免多精灵同屏叠加超预算。
- 全程不透明 `plain()` 色块,无渐变、无 alpha,与现有 chain_lab 视觉风格(表盘/光点本就是
  纯色块+描边)一致延续。

## 11. 开放风险(留给实现前/实机验证时决策)

1. **摇杆 Y 轴 v1 不用**:是否要在快速迭代里加第二排战利品(用 Y 轴控制前后排)——建议先把
   一维版本做扎实、实机跑通后再评估要不要加,避免第一版就要求孩子同时应付"两轴定位+拧旋钮
   下降"三种精细动作。
2. **编码器"拧曲柄"对幼儿精细动作可能偏难**:拧到精确格数比倾斜/覆盖/按压更依赖手指精细
   控制。已有的缓解手段是 §5 的"对齐容差"+ §9 的"接近最大深度自动到底容忍",但具体好不好玩
   需要实机验证,不好玩的话可能要重新设计成"编码器只控制方向、深度自动匀速下降"这类更简单的
   映射(留作后续迭代方向,本轮先按"曲柄"设计,实机验证后再定)。
3. **诊断模式切换方案 A vs B**(§7):运行时双键长按 vs 编译期开关,实现时二选一,建议先做
   B(零风险),A 作为体验加分项延后。
4. **战利品主题/造型本轮不展开**:留给下一步创作,建议照 peekaboo 访客表(`apps/peekaboo/
   SPEC.md` §5.1)的列格式(id/主色/造型要点/音签/出场动画)逐个设计,与"抓娃娃机"主题
   (玩具/小动物/糖果均可)保持后续再定,不是本轮的结构性缺口。
5. **两节点级联在孩子真实用力摇/转下的链路稳定性**:现状 README 已标注"Core2 直连 Chain
   host 未经官方背书",诊断台阶段的测试强度偏温和;真游戏会有更持续/更用力的物理输入(反复
   拧曲柄、推摇杆到底),建议先做一次"正常玩法强度"下的连接稳定性抽查(哪怕只是先跑 §11①的
   一维简化版实机盯一会儿),确认链路顶得住再投入完整游戏层的精雕细琢。
