# chain_lab — 抓娃娃机(as-built)

> 本文件是 **chain_lab 应用的竣工记录**;玩法规格见 `SPEC.md`(重写规格,完工后保留)。
> 跨应用平台事实(桌面玩法省电坑)见根平台手册 `CLAUDE.md`(§10 做新 App 指南、§7 电源·休眠、§11 坑位)。
> **占 ota_4(0x990000)**。烧录:`tools/flash_one.sh chain_lab`(= esptool write-flash 0x990000)。

产出:在已跑通的 Chain UART 传输层(`chain_bus` + `unit_chain_encoder` + `unit_chain_joystick`)
之上盖了一层「抓娃娃机」——摇杆推吊臂、编码器像摇曲柄一样降爪子、按下抓取、战利品收进展示架、
集齐一批开派对。原本的仪表台(左圆环表盘 + 右光点方框)折叠成隐藏诊断模式,排障能力不丢
(见下「隐藏诊断模式」)。守零失败/即时因果/多通道冗余/渲染红线。

**v2.1(2026-07-10)深度分层迭代**:首版实机试玩反馈"深度纯仪式、按键可有可无"(转到底自动抓
+ 成败与深度无关,整局唯一真实决策只有横向对齐)。本版把深度改成**选择器**:战利品分住浅/中/深
三个深度层(高低底座),抓取成败 = 横向对齐 + 深度层匹配双条件,详见下「状态机」。
**v2.1 已实机验证通过**(分层选择/碰到反馈/dwell 兜底可用)。

**v2.2(2026-07-10)趣味批**(仿 busy_knobs 趣味增量批次先例):①战利品从色块升级为有脸玩偶
(熊/呆毛/小鸡/青蛙/兔,`PRIZE_LOOK` 造型表,SPEC §11④ 落地);②爪子改横梁+两指开合造型;
③拧曲柄每格轻"咔"(SPEC §4 反馈矩阵欠账,音调随深度下沉,节流防噪);④**金星彩蛋**:正常
战利品收进展示架后 33% 概率在某个空坑位的**最深层**冒出金星,抓到触发小庆祝(金色闪+琶音+灯带
扫圈)、不占展示架不参与集满——给"转到最深"一个主动追求的理由。摇杆 Y 轴第二排(SPEC §11①)
**明确不做**:深度分层已占据第二决策维度,再叠第三个空间维度会回到"三种精细动作叠加"的担忧,
且平面渲染难以向不识字幼儿传达前后排。

## 协议来源(逐字节核实,协议层本轮零改动)

官方 PDF 只给链接不含正文;帧格式**从官方 Arduino 库 `github m5stack/M5Chain` 源码逐字节核实**(AGENTS.md §1 的"MCP 缺板级细节 → 厂商 GitHub raw"退路)。已回填进 `docs/units/Chain_Encoder.md`/`Chain_Joystick.md` §4。

帧 = `AA55 | lenLo lenHi | id | cmd | data | crc | 55AA`,`len=3+dataLen`(小端)、`crc=(id+cmd+data)&0xFF`、`id`=链上位置(1 起);应答载荷小端。节点主动发心跳(0xFD~1/s)/枚举(0xFC)/按键上报(0xE0),主机轮询按 (id,cmd) 匹配、其余丢弃。

## 平台新增(可复用,本轮未改动)

- ①`components/units/chain_bus` —— Chain UART 传输层:装 UART2(G14 TX/G13 RX)、`chain_bus_request(id,cmd,data..)` 一次请求/应答,通用命令(设备类型/固件版本/RGB/亮度),`chain_bus_sniff(ms)` 抓原始字节做诊断;
- ②`components/units/unit_chain_encoder`(U207,GET_VALUE=0x10 绝对计数 int16 + BUTTON 0xE1);
- ③`components/units/unit_chain_joystick`(U205,GET_16ADC=0x30 原始 ADC + BUTTON 0xE1,应用侧软件归中)。

## 抓娃娃机玩法(as-built)

### 架构:传输层 / 游戏层拆分

- `chain_lab.c` 收窄为纯"传输/绑定层":`scan_bus`/`poll_enc`/`poll_joy`/`node_rgb`/`hue2rgb`/
  `joy_calibrate_center` 原样保留(SPEC §1 红线),`game_task` 20Hz 轮询节奏 + `core2_sleep`
  集成 + 深度省电重扫原样保留。新增一组 getter/RGB 包装(`chain_lab_enc_value()` 等,见
  `chain_lab.h`)把读数暴露给游戏层,游戏层不碰协议细节。
- `crane_game.c/.h` 新增:状态机、吊臂/爪子精灵、战利品坑/展示架、派对编排,只消费
  `chain_lab_*` getter。
- **SPEC §8 文件布局的一处落地差异**:`diag_view.c/.h` 未单独建文件,采用 SPEC §7 明确给出
  的备选做法——诊断台 UI(`ui_create`/`ui_enc`/`ui_joy`/`ui_status`)整段留在 `chain_lab.c`,
  用 `tuning.h` 的 `CHAIN_LAB_DIAG_MODE` 编译期宏包起来,和游戏层调用点(`chain_lab_start`/
  `game_task`)二选一分支。两种模式都已过编译验证(见「状态」)。

### 状态机(SPEC §3 落地)

`PLAY_IDLE ⇄ DESCENDING → GRABBING(~500ms)→ ASCENDING(~750ms)→ DEPOSIT(~500ms)→ PLAY_IDLE`,
展示架集满 5 种 → `PARTY`(~3.5s)→ 重置换新一批 → `PLAY_IDLE`。

- 摇杆 X 直接映射吊臂横坐标,**任何状态下都可移动**(不受深度状态门控,SPEC §3);
- 编码器帧间 delta 只在 `PLAY_IDLE`/`DESCENDING` 驱动深度(顺时针+/逆时针-,到 0 回
  `PLAY_IDLE`);接近底部(`DESCEND_MAX_PX - DESCEND_SNAP_TOL`)自动到底并直接判抓(兜底不变);
- **深度分层(v2.1)**:每轮 5 个战利品随机分住 `PIT_LAYERS=3` 个深度层(浅/中/深各至少一个),
  层目标深度 = `DESCEND_MAX_PX - (2-层)*PIT_LAYER_STEP_PX`(52/96/140px),战利品下垫高低底座、
  最深层贴地无底座;
- **抓取成败 = 双条件**:横向对齐(`GRAB_ALIGN_TOL_PX` 中心容差带)**且**深度落在该层带
  (层目标深度 ± `GRAB_DEPTH_TOL_PX`)——拧曲柄从纯仪式变成"选哪一层"的真实决策;
- **碰到反馈("对上了")**:双条件同时满足的瞬间,战利品小幅摆动 + `SND_NEAR` 上扬叮铃 +
  轻震 + 编码器节点灯变该战利品主色 + 底座灯带 `LED_BASE_NEAR` 加亮——"现在收爪就是它";
- **三种收爪方式**:①抓取键任意时刻按下(在层带内 = 抓中,带外 = 空爪"诶?");②碰住且
  不再转曲柄持续 `TOUCH_DWELL_MS`(700ms)自动收爪(不会按键的孩子的兜底,转动会重计时);
  ③转到底吸附自动收爪(沿用 v2);
- `GRABBING`/`ASCENDING`/`DEPOSIT`/`PARTY` 期间编码器按键/深度输入被状态门控吃掉(不排队、
  不触发二次动画),摇杆横移与摇一摇彩蛋不受影响。

### 反馈矩阵(SPEC §4 落地要点)

六类事件(移动/下降/抓取/抓中/抓空/集满)×六条通道全部接上;v2.1 新增第七类事件
**碰到("对上了")**:战利品摆动(屏)+ `SND_NEAR`(音)+ `HAPTIC_BUMP_LIGHT`(震)+
`LED_BASE_NEAR`(灯带)+ 编码器节点灯变战利品主色(节点 RGB)——底座灯带的 NEAR 语义从
v2 的"下降中"收窄为"碰到了",更贴近 tilt_maze 的接近目标用法。**抓空是"诶?"轻弹一下**,
不掉血、不计失败、零负面反馈(不用红色,用中性琥珀)。摇杆节点 RGB 平时沿用现状 `nx/ny→r/b`
映射,派对期间与编码器节点同步跑彩虹(`hue2rgb`)。

**一处降级(非结构性)**:SPEC §4"下降"行要求底座灯带"暖色微调暗一档",现状 `ledstrip_fx`
没有对应的第三档基础模式;落地时复用了已有的 `LED_BASE_NEAR`("接近目标:加亮向目标色")
代替——语义仍贴合("正在下降接近战利品"),没有为这一处观感新增组件 API。

**v2.2 补上 SPEC §4"下降每格轻咔"**:拧曲柄每格合成一声 20ms 短音(`audio_fx_play_notes`
单音符),音调随深度线性下沉(1500→940Hz,声音本身就传达"下到多深"),`CRANK_TICK_MIN_MS`
(150ms)节流防连转刷屏噪音。

### 战利品玩偶(SPEC §11④,v2.2 落地)

`PRIZE_TYPES=5` + 金星彩蛋(`KIND_GOLDEN=5`),造型由 `crane_game.c` 的 `PRIZE_LOOK` 表驱动
(身体圆 + 特征件 A/B + 双眼 + 嘴,全部 plain 色块,零 alpha):

| kind | 主色 | 造型 | 特征件 |
|---|---|---|---|
| 0 | 珊瑚红 | 小熊 | 圆耳×2(脑后) |
| 1 | 天蓝 | 呆毛 | 头顶单圆 |
| 2 | 暖黄 | 小鸡 | 红冠 + 橙喙 |
| 3 | 草绿 | 青蛙 | 白凸眼×2(瞳在上) |
| 4 | 紫粉 | 兔子 | 长耳×2 |
| 5 | 金黄 | 金星(彩蛋) | 侧光球×2 |

实现:每个坑位一个**透明容器**(整只一起移动/隐藏,带 `DOLL_HEADROOM` 头饰空间与
`DOLL_PAD_X` 防裁边距),零件按 kind 每轮重新装扮(`pit_slot_paint`),渲染代价只在换轮/
碰触摆动时付,与 v2.1 色块同量级。被抓跟随精灵 = 身体+双眼(不带特征件,简化)。
**金星机制**:正常收获后 `GOLDEN_CHANCE_PCT`(33%)概率在随机空坑的最深层出现(同屏最多
一颗),音签用 `SND_NEAR` 提示出现;抓到 → 飞向展示架正中 + 三连琶音 + `LED_FX_SWEEP_L2R` +
两节点金色,不占展示架、不参与集满,换轮自然消失。

### 电源 / 省电

kick 判据复用 `poll_enc`/`poll_joy` 现有 `activity` 返回值(**`poll_joy` 的判据 2026-07-14
重做过,见下节「摇杆回中偏置」**)。深度省电唤醒后除了原有的"断电重扫"分支,新增
`crane_game_reset_position()`:爪子收起、吊臂回中、正在进行的抓取(若被打断)战利品还给坑
不计入收集,并重同步编码器基准(避免节点复位计数清零被误读成一次大幅转动)。

## 🔴 摇杆回中偏置(2026-07-14 实机标定,踩坑记录)

**现象**:`poll_joy` 原来拿"绝对偏移超 `JOY_MOVE_KICK`(0.15)"当"有人在玩"的 kick 判据。
但这颗 Chain 摇杆松手后的静止点并不回到开机校准中心 → 偏移恒定非零 → 每帧 kick →
**永不打盹、永不深度省电**,500mAh 电池被 70% 亮度常亮耗干。这也是验收清单里
"打盹/深度省电如常进出"一直没打上勾的那一项(从没验到过)。

**实机日志实测**(临时开 `JOY_CAL_LOG`,20Hz 采样,开机校准 `c=(31434,33599)`):

| 阶段 | 静止时原始 X | 相对校准中心偏移 |
|---|---|---|
| 开机后静置(没碰过摇杆) | 31434~31444 | ≈ **0.000** |
| 推到最左、松手后静置 | 31207~31223 | **+0.147** |
| 推到最右、松手后静置 | 31206~31262 | **+0.13~0.15** |

**根因**:不是校准时机不准(开机校准很准),也不是滞环(推左推右偏到**同一个**新点)——
是**摇杆被推过一次后机械回中点就永久偏了约 220 ADC(≈0.15,峰值 0.21),之后 100 多秒
都不回来**。所以"把开机校准做对"根治不了:孩子玩第一下,偏置就回来了。静置噪声地板
(帧间 ADC 变化峰值)基线 4~24、被推过之后升到 **43**。

**定案改法**(`tuning.h` + `chain_lab.c: poll_joy()` + `crane_game.c`):

- kick 主判据换成**帧间 ADC 变化量** `JOY_MOVE_ADC=80`(噪声地板 43 的近 2 倍余量),
  恒定偏置够不着 —— 同 `CLAUDE.md` §7 用 IMU 帧间差判"有没有人在玩"的思路。
- 副判据 `JOY_HOLD_KICK=0.35`:位置型玩法下孩子会**推着杆保持瞄准**(帧间不变),偏移显著
  时也算在玩;0.35 远高于实测最大回中误差 0.21,不会被偏置顶穿。
- **自适应回中** `JOY_RECENTER_BAND=0.25` / `JOY_RECENTER_PCT=1`(τ≈5s)修掉那 0.15 偏置。
  🔴 **参数和门控都比 busy_bus 收紧,别照抄**:busy_bus 是速度型驾驶(偏移→速度),中心被
  慢慢吃掉只表现为"轻推着会减速";本作是**位置型**(`s_arm_x = 屏幕中心 + nx × 半行程`),
  中心被吃掉 = **吊臂自己滑回屏幕中间**,孩子瞄准的娃娃会跑掉。因此除了收紧参数,还加了
  状态机门控 `crane_game_recenter_ok()`:**只在 `PLAY_IDLE` + 爪子在顶时才校正**,一进
  下降/抓取/上升/落架/派对就冻结中心。
- 吊臂加死区 `JOY_ARM_DEADZONE=0.04`(死区外线性重缩放,无跳变):盖住 ±40 ADC 噪声造成的
  ±3px 吊臂抖动,松手后吊臂稳稳停住。

**排障设施**:`tuning.h` 的 `JOY_CAL_LOG=1` 打开后每 0.5s 打一行
`joy raw=… c=… n=… dmax=…`(原始 ADC / 校准中心 / 归一化 / 帧间变化峰值),摇杆再出漂移
类问题先开它拿实测曲线,**别凭猜测调死区参数**(busy_bus 就是猜参数白花了一轮刷机验证)。

## 验证台历史背景(v1,保留)

本 App **头一次接 Chain 系列外设**——不是 Grove I2C 而是 **UART 115200 8N1 菊花链**
(Core2 PORT.C 直接作 Chain host)。v1 是一张纯功能验证台:左圆环表盘(指针角=编码器绝对
计数)+ 中心数值;右方框光点(位置=摇杆归一化 X/Y)+ 原始 ADC 数值;按键 → 光点/指针变绿 +
节点板载 RGB 闪白 + 轻震;平时编码器灯随计数走彩虹、摇杆灯随方位换色——同时验证了读(RX)
与写节点 RGB(TX)两条路,证明 Core2 直连 Chain host 这条路线可行。这套 UI 现在整段保留在
`CHAIN_LAB_DIAG_MODE=1` 分支里,不是被删掉了。

## ⚠️ Core2 直连 Chain host 未经官方背书

硬件文档标注未验证,官方范式要挂独立 Chain 主控(如 DualKey)。诊断模式下 `chain_bus_sniff`
仍可抓 PORT.C 原始字节自诊:有心跳=链路通只是没认到,一字节都无=没供电/接反/直连不成立。

## 🔴 供电与省电

- **PORT.C 供电 = M-Bus 5V(EXTEN)**,与 PORT.A 单元同源(`core2_board_init(enable_leds=true)` 已代开);深度省电切 5V → 节点断电复位,唤醒后重扫接管 + 游戏层复位安全位置。
- **桌面玩法省电坑**照旧:机身不动≠没人玩,转/推/按都 `core2_sleep_kick`。
