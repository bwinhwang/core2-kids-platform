## 0. 为什么重写(诊断)

v1 的问题和 peekaboo v1 同源:**单一维度被收窄成一个开关**。

1. **手势只有一种**:靠近 = 喂,远离 = 不喂。超声波给的是连续距离,但唯一被读取的信息是
   "是否 ≤ 90mm",连续的"嘴张合 + 音高"(空气琴层)很讨喜,但从不影响任何结果——好听但和
   "玩法"无关。
2. **被喂的对象只有一个**:永远是同一只薄荷绿怪兽,永远是饼干,永远同一套动画/音效。
3. **没有积累**:喂满 5 个 → 全屏庆祝 → **清零重开**,下一轮和这一轮没有任何区别。3~4 岁
   最吃"重复但有变化"(again! but different),v1 只有重复没有变化。
4. **没有悬念/多样反应**:不管怎么伸手靠近,结果都一样——猛地伸手 vs 轻轻探过去,游戏毫无
   察觉。

## 1. 核心概念:「手势食堂」

同一根超声波距离信号,像 peekaboo 从"照度"里多挖"松开速度"和"捂的时长"一样,**多挖三条
手势信息**,再配一套"轮流来访的小客人 + 相册收藏"撑住重复可玩性:

- **扑喂**(快速欺近):手快速逼近吃区 → 夸张咬一口,适合调皮型食客。
- **抚摸**(靠近后停留):进吃区后不走 → 追加一段"蹭蹭"反应,适合怕生型食客。
- **挠痒**(中距离来回摆动,不进吃区):不算喂食,纯彩蛋——逗当前在场的角色笑一下。
- **主怪兽常驻**为"店主",不被吃客顶替;没有客人在场时喂食目标退回主怪兽(即 v1 原逻辑
  兜底,永远有"人"可以喂,不会有空场感)。客人在场时,喂食目标转为客人,主怪兽在旁陪着开心。
- **食客轮流拜访**,喂满次数就满足离场,**新物种**离场时点亮相册;集满相册 → 全员茶会大庆祝
  → 相册清灰、下一轮重新抽签。

零失败不变:三种手势没有"对错",喂错偏好只是反应小一点,依然计入进度、依然是正反馈。

## 3. 状态机

### 3.1 大局状态

```
ST_PLAY(常态主循环,含访客子状态机 §3.2)
   ──(相册集满 ALBUM_SIZE 位 & 当前访客已离场)──► ST_PARADE(~3.2s 全员茶会)
   ──(收场:相册清灰、抽签池重置)──► ST_PLAY
```

### 3.2 访客子状态(嵌在 ST_PLAY 内)

```
GUEST_NONE ──(拜访计时到 & AWAKE & s_unit_ok)──► GUEST_ARRIVING(~600ms 入场)
GUEST_ARRIVING ──► GUEST_PRESENT(可反复喂,满 FEEDS_PER_GUEST 次判定满足)
GUEST_PRESENT ──(满足)──► GUEST_LEAVING(~500ms 开心离场;新物种先点亮相册)──► GUEST_NONE(重摇拜访计时)
```

`GUEST_NONE` 期间喂食目标 = 主怪兽(v1 原饼干循环,原样保留作兜底,**不占相册/不计
`FEEDS_PER_GUEST`**);有客人在场时喂食目标切到客人,主怪兽只在旁做"陪玩"弹跳反应
(复用现有 `monster_bounce`,幅度调小)。

### 3.3 打断与边界(必须全覆盖,仿 busy_knobs 小鸟表)

| 事件 | 访客处置 |
|---|---|
| `enter_parade()` | 立即 `GUEST_LEAVING`(不等满足;全员登场是 Parade 自己的编排,见 §9) |
| 进 NAP / DEEP | **瞬时隐藏**(HIDDEN + 状态置 `GUEST_NONE`,不放动画——省电态不许跑动画) |
| 唤醒 | 保持 `GUEST_NONE`,拜访计时重新随机 |
| `unit_lost()`(拔线) | 立即隐藏回 `GUEST_NONE`;`s_unit_ok==false` 期间不再拜访 |
| DEEP 恢复后 `unit_attach` | 无特殊处理(访客状态已是 `GUEST_NONE`) |

## 4. 手势判定(核心算法,同一根超声波挖三条信息)

沿用现有 `on_reading(mm, stage)` 的读数节拍(每 `SONIC_READ_TICKS` 帧一次,约 99ms/读)。

### 4.1 扑喂 / 抚摸(在 `EAT_MM` 边沿触发点分类,零额外延迟)

边沿触发本身**不变**(`s_dist <= EAT_MM && s_armed` 立即 `do_eat()`,<100ms 即时因果不受影响),
只是分类用的是**已经算好的历史数据**,不需要等待:

```c
float delta = s_dist_raw_prev - mm;   // 正 = 正在靠近,单位 mm/读(~99ms)
gesture_t g = (delta >= APPROACH_FAST_MM_PER_READ) ? GESTURE_POUNCE   // 扑喂
                                                    : GESTURE_NORMAL;  // 普通/慢速靠近
do_eat(g);
```

**抚摸是叠加在触发之后的追加判定**,不阻塞 `do_eat()` 的即时反应:

```c
// 每读:距离仍在吃区内则计数,离开则清零(与 s_armed 的 EAT_EXIT_MM 迟滞共用同一次"退出"判据)
if (s_dist <= EAT_MM) {
    if (++s_hold_reads == PET_HOLD_READS && !s_pet_fired) {
        s_pet_fired = true;
        do_pet_bonus();      // 追加"蹭蹭"反应,不是新的一次喂食,不重复计 FEEDS_PER_GUEST
    }
} else {
    s_hold_reads = 0;
    if (s_dist > EAT_EXIT_MM) s_pet_fired = false;   // 与 s_armed 同一退出边界复位
}
```

### 4.2 挠痒(中距离来回摆动,`s_dist > EAT_MM` 时才跑,与喂食判定互斥避免打架)

**泄漏计数**,与 busy_knobs 摇一摇(`SHAKE_THRESH`/`NEEDED`/泄漏机制)同一套路,只是信号换成
"方向反转"而非"加速度突变":

- 每读算 `delta = s_dist_raw_prev - mm`;`|delta| < WAVE_NOISE_MM` 视为抖动噪声,不参与方向判定。
- 维护当前方向段累计位移 `s_wave_run_mm`;方向翻转且 `s_wave_run_mm >= WAVE_SEG_MM`(防抖动
  误判成"来回")才算一次**有效反转**:计数器 `s_wave_hits` 加 1(封顶 `WAVE_REVERSALS_NEEDED`);
  方向未翻转或翻转但位移不够,计数器不加。
- 每读若本次没有产生有效反转,计数器泄漏 -1(下限 0)——要连着像模像样地摆几下才算数,单次
  手抖不会误触发。
- `s_wave_hits >= WAVE_REVERSALS_NEEDED && cooldown==0 && s_dist > EAT_MM && AWAKE && ST_PLAY`
  → 触发挠痒:计数器清零,冷却 `TICKLE_COOLDOWN_MS`。

### 4.3 反应对象

三种手势的反应对象统一是"当前在场目标"(`GUEST_PRESENT` 则客人,否则主怪兽)——即使
`GUEST_NONE` 兜底态,也能扑喂/抚摸/挠痒主怪兽,手感提升不依赖客人是否在场。

## 5. 摇一摇彩蛋(IMU,P3)

直接复用 busy_knobs 已验证有效的数值和泄漏机制(`SHAKE_THRESH=1.2f` / `SHAKE_NEEDED=3` /
`SHAKE_COOLDOWN_MS=1500`),信号也是同一个"帧间三轴加速度变化和"——**不需要重新调参**。
触发效果:当前在场目标(客人或主怪兽)"晕乎乎"摇一下(x ±3px 抖动 3 次,~250ms)+ 一声滑稽
短音 `{330,50,40}{294,50,35}` + `HAPTIC_BUMP_LIGHT`。纯装饰彩蛋,不影响进度、不计入
`FEEDS_PER_GUEST`。因为本身就是"在玩"的信号,允许 `core2_sleep_kick`(与现有摇晃可唤醒逻辑
自然叠加,不需要额外处理)。

## 6. 访客表(P1,5 常驻物种)

全部 `plain()` 不透明色块拼装,容器 ~44×40("小客人"体量,不喧宾夺主);预建 HIDDEN,
到访时显示。站位:客人站在主怪兽右侧地面 `x≈220, y=GROUND_Y-容器高`(脚贴地,不挡脸)。
颜色为起步值,RGB565 实显可微调。

| PIGGY | 小猪 | 粉 `0xFFB3C6` | 圆鼻+卷尾+圆身 | 扑喂 | {392,80,60}{330,100,55} | 摇摆跑入 450ms(x 快+y 小抖) |
| TURTLE | 乌龟 | 绿 `0x7FB069` | 龟壳纹(3 弧线)+短腿 | 抚摸 | {294,140,45}{330,160,40} | 慢挪入场 900ms |

## 8. 优先级 / 冲突矩阵(结算顺序)

1. **相册集满 → `enter_parade()`** 最高优先,命中即 return;当前访客立即 `GUEST_LEAVING`。
2. **喂食边沿触发**(§4.1):`s_armed && s_dist<=EAT_MM` 立即 `do_eat()`,速度分类只读历史、不
   加延迟。
3. **抚摸追加**(§4.1 hold 计数)与**挠痒**(§4.2)互斥:抚摸只在 `s_dist<=EAT_MM` 时跑,挠痒只在
   `s_dist>EAT_MM` 时跑,同一时刻只有一个在计数。
4. **摇一摇彩蛋**(§5):与喂食/挠痒判定完全独立(信号源是 IMU 不是超声波),可能与喂食同帧
   触发,两者反馈各自播放,音频队列自然排开(≤8 音/~400ms 限制下可能截断次要音,可接受)。
5. **省电**:访客子状态机 tick 只在 `AWAKE` 跑;进 NAP/DEEP 访客瞬时隐藏,不参与上述任何判定。

## 9. 相册收藏 + 全员茶会(Parade)

- 屏幕顶部一排 `ALBUM_SIZE=5` 枚头像位(仿 peekaboo `album.c`,同一实现模式:未收集=灰底
  剪影,收集=点亮真色);客人首次满足离场时点亮对应头像。
- 集满 5 位 → `ST_PARADE`(~`PARADE_HOLD_MS`=3200ms):5 位已收集客人依次从两侧滑入排成一排
  (每位间隔 ~150ms 错峰,复用 busy_knobs `wave_dir_t` 错峰手法)+ 主怪兽居中大弹跳 +
  庆祝迸发(复用现有 `enter_win` 的 `BURST_COUNT` 迸发框架,规模不变,只是触发时机改为
  "集满"而非"喂满 5 个饼干")。
- 收场:相册清灰、访客抽签池重置(下一轮首抽不与本轮最后一位离场客人重复,防"刚送走又是它",
  规则同 peekaboo §5.2)、回 `ST_PLAY`。
- (P4)NVS:仿 peekaboo `album.c` 原样复用模式(`nvs_get/set_u8` 相册位掩码 + `u16` 茶会累计
  次数),命名空间 `"feed_monster"`,键名 `album_mask` / `parade_count`。分区表已有 `nvs`
  分区,不改表。

## 10. `tuning.h` 新增常量(集中一处,默认值如下;现有 v1 常量原样保留)

```c
// ── 访客系统 ──────────────────────────────────────────────────────────
#define ALBUM_SIZE            5      // 常驻物种数(相册头像位数)
#define FEEDS_PER_GUEST        3      // 每位客人喂几次算满足离场
#define GUEST_VISIT_MIN_S      15     // 拜访间隔(s,随机区间;比 busy_knobs 小鸟短,因为是核心玩法)
#define GUEST_VISIT_MAX_S      30
#define GUEST_ARRIVE_MS        600    // 入场动画时长
#define GUEST_LEAVE_MS          500    // 离场动画时长
#define RARE_PCT                8      // (P4)彩虹鱼替换本次抽签的概率(%)

// ── 手势判定:扑喂 / 抚摸 ─────────────────────────────────────────────
```

## 11. 渲染预算核对

- 常态帧:嘴高度(现有,~2ms 级脏矩形)+ 访客单个小容器位移(≤44×40≈1,760px,一帧内单次动画
  更新,远低于 §6.2 的 ~15,000px/帧预算)+ 挠痒抖动/摇一摇抖动(同量级,且触发频率低、互斥
  不叠加)。
- 到访/离场动画期间(<1s)瞬时叠加量仍是个位数千像素,不接近预算上限。
- Parade 5 位错峰滑入 + 迸发星,量级与现有 `enter_win`(`BURST_COUNT=8`)相当,已验证可接受,
  帧率按 §6.5"庆祝彩纸"档位(15~20fps 可接受)处理。
- 全程无 alpha 混合、无渐变;相册头像用现有 `plain()` 色块切换(灰↔真色),非动画属性直接
  set_style,零动画开销。

## 13. 分阶段优先级(裁剪范围用,P1 是能解决"单调"问题的最小集)
