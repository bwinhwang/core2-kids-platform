# MCU 固件从机型 I2C 单元 —— 驱动读写参考

> 适用范围:**从机是 MCU(跑固件)** 的 I2C 单元(如 M5 8Encoder / STM32 固件那类 Base/Unit)。
> 它们的读时序不同于 AXP192 / MPU6886 / FT6336 这类**硬件寄存器芯片**,接入时按下列通则处理。

## 1. 寄存器读必须两笔事务(最关键)

- 读寄存器 = 先「写寄存器号 + STOP」,再「单独发起读」**两笔独立事务**。
- **禁止 repeated-start 组合读**:IDF 的 `i2c_master_transmit_receive`、`i2cget` 默认读法、
  Arduino `requestFrom` 不带 stop 等,都是组合读。
- 原因:这类固件通常**只在收到 STOP 时**才解析寄存器号、备好回读数据。组合读在读方向的地址阶段
  到来时从机数据尚未就绪,会拿着 `tx_len=0` 进发送态 → **无限拉伸 SCL 钳死总线,断电才恢复**
  (固件自带的超时自恢复往往解不开发送态拉伸)。
- 官方 Arduino / UIFlow 库都是「写完 STOP 再读」,照此实现即可。

## 2. 在位检查必须做一次真正的寄存器读

- **不要只用地址 probe**(`i2cdetect` / 单纯 ADDR+ACK)判断单元在位或健康。
- 地址 ACK 是外设**硬件**行为:即便固件读法不兼容、甚至固件已卡死,地址仍会 ACK。
  `i2cdetect` 看得到 ≠ 能正常读。
- 应读一个已知寄存器(如固件版本号)并校验返回值。

## 3. 先核对事务框架,再照抄寄存器表

- 写驱动前,先看官方宿主库**每笔事务的线上形状**(START / Sr / STOP、方向、长度),而不仅是寄存器地址表。
- 对 MCU 固件从机,事务的形状(有没有 STOP)与寄存器表同等重要,同属通信契约。

## 4. 症状识别

- **每次都死在同一笔事务**(通常是第一笔寄存器读)、随后整条总线被钳死需**断电才恢复** =
  软件读法问题的签名,**不是硬件坏**。
- 遇此先改读法(规则 1),再去怀疑供电 / 上拉 / 换货。

## 5. 范围界定

- **硬件寄存器机**(AXP192 / MPU6886 / FT6336 等)不受此限,可用 repeated-start 组合读。
- 只有「从机是 MCU 固件」的单元才需规则 1~4。
